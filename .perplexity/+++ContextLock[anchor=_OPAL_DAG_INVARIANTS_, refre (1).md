## Part III: Pattern Analysis — Structural Findings

### 3.1 Pattern 7.1 — Node-Level Epistemic Isolation

**Validated Status: CONFIRMED with qualifications.**

The Opal Generate node, when equipped with multiple tools simultaneously (e.g., `@Search Maps` AND `@Use NotebookLM`), exhibits Mixture-of-Experts behavior without any explicit routing arbitration logic. The Gemini model backend must perform implicit schema reconciliation across heterogeneous tool outputs in a single generation pass. The Console debug tool (revealed in the official documentation) provides the only native visibility into this process — it exposes model inputs, tool calls, thinking steps, and tool outputs in real-time.[^1_5]

**Architectural Mitigation — Tool Isolation Protocol:**
The +++MereologyRoute principle is operationalized in Opal by **separating tools into dedicated single-purpose Generate nodes** and chaining them sequentially via step references. Each node becomes a bounded epistemic regime with a single knowledge modality:

```
WRONG (Multi-Tool Pollution):
Generate Node 1: Tools = [@Search Maps, @Use NotebookLM, @Search Web]
→ Gemini must reconcile GeoJSON + RAG prose + live search in one pass
→ Attention dilution → schema conflict → hallucination risk

CORRECT (Tool Isolation via MereologyRoute):
Generate Node A: Tools = [@Use NotebookLM] only    ← ANCHOR
Generate Node B: Tools = [@Search Web] only         ← AUGMENT (refs A)
Generate Node C: Tools = [@Search Maps] only        ← LOCATE (refs B)
Generate Node D: Tools = none (refs A, B, C)        ← SYNTHESIZE
```

This forces the Gemini model to operate in a single semantic modality per node, reducing attention dilution and making the Console trace interpretable for debugging.[^1_5]

### 3.2 Pattern 7.2 — NotebookLM Epistemic Anchoring

**Validated Status: PARTIALLY CONFIRMED — architectural constraint identified.**

The NotebookLM integration in Opal (experimental as of February 2026)  provides what this research designates a **+++ContextLock** — a synecdochic anchor that constrains the Generate node's generative range to a curated, human-validated document set. The geeky-gadgets.com demonstration (March 2026) confirms that NotebookLM notebooks integrated with Opal can produce "detailed reports based on user queries, personalized action plans tailored to specific goals, and audio summaries" — all grounded in source material rather than parametric hallucination.[^1_6][^1_7][^1_8]

**Critical Architectural Constraint — Static Snapshot Problem:**

The Google Cloud NotebookLM Enterprise documentation reveals a fundamental epistemic limitation:[^1_5]

> *"When you add a document as a source to a notebook, a copy of the document is added to the notebook... Changes to the original document aren't reflected in the copy."*

This means that in any long-running Opal workflow (e.g., a 50-step Agent Mode loop running over days), the NotebookLM grounding context becomes **increasingly stale relative to the live world**. The `Semantic Drift Score (SDS)` — the divergence between NotebookLM source truth and real-world state — grows monotonically as the workflow ages, regardless of what the Web Search tool retrieves. This is not a bug; it is a structural property of the source-copy architecture.

**Mitigation — Versioned Snapshot Protocol:**

1. Pin NotebookLM notebooks to **explicit version epochs** (e.g., "v2026-Q1-snapshot")
2. Create a dedicated `Generate Node [Conflict Detector]` that takes both NotebookLM output AND Search Web output and explicitly prompts: *"Identify any factual contradictions between [NotebookLM_output] and [SearchWeb_output]. Return a JSON list of conflicts with confidence scores."*
3. Route conflict-flagged outputs through the FIPI Code Execution validator before proceeding

### 3.3 Pattern 7.3 — Saga-Style Recovery via Code Execution

**Validated Status: THEORETICALLY SOUND — sandbox constraints require engineering.**

Opal's Code Execution tool is documented as a utility node in the tool vocabulary. The proposed FIPI (Failure-Informed Prompt Inversion) pattern repurposes it as a **compensating transaction trigger** in the Saga orchestration model. The mechanism works as follows:

The Code Execution node receives the JSON output of the previous Generate node as an injected reference (via `@` step reference). A Python script then performs:

1. **Structural validation**: Does the output conform to the expected JSON schema?
2. **CFDI calculation**: $$
\text{CFDI} = \frac{|\text{Confidence}_{\text{logits}} - \text{Fidelity}_{\text{AST}}|}{\text{Token\_Depth}}
$$
3. **Semantic consistency check**: Do key claims in the output contradict the NotebookLM anchor?
4. **Routing decision**: If CFDI < 0.15, pass; if CFDI ≥ 0.15, trigger compensation
The **compensation transaction** is implemented via Opal's `Go to...` routing node, which redirects execution back to the NotebookLM anchoring layer with an injected `failure_context` object describing the specific failure mode. This is the structural analogue of a Saga rollback.[^1_2]

**Sandbox Constraint Warning:** The Code Execution sandbox within Opal operates with restricted scope — external network calls, file system access, and long-running processes are constrained. Python scripts must be stateless, operate on injected string/JSON inputs only, and complete within the sandbox's latency budget. The FIPI script must be kept to pure Python with no external imports beyond `json`, `ast`, and `re`.

## Part IV: Threat Model Analysis

### 4.1 Hypothesis 1 — The Autophagic Memory Trap

**Threat Classification: CRITICAL — Confirmed Architecturally Plausible**

If an Opal workflow activates both Agent Mode's `Use Memory` (Google Sheets backend)  and `Use NotebookLM` simultaneously, within an autonomous loop, the following failure cascade becomes structurally possible:[^1_2][^1_3]

```
AUTOPHAGIC MEMORY TRAP — FAILURE CASCADE
Step 1: Agent Step reads from Memory (Sheets) → retrieves previous session output
Step 2: Agent Step uses NotebookLM → generates response conditioned on Memory
Step 3: Agent Step writes synthesized output BACK to Memory (Sheets)
Step 4: Next iteration reads the hallucinated synthesis as factual prior knowledge
Step 5: NotebookLM grounding is now contaminated by hallucinated context
Step 6: Loop continues — each iteration drifts further from source truth
Step 7: Output is maximally confident, perfectly formatted, completely fictional
```

This is not hypothetical — it is the direct mechanical consequence of the Agent Step's memory write-back architecture: *"It autonomously selects tools like Google Sheets for maintaining memory across different sessions."* The agent has no native self-audit mechanism to distinguish between "memory I wrote based on verified facts" and "memory I wrote based on a previous hallucination."[^1_3]

**Mitigation — Memory Epoch Tagging Protocol:**

Every write to the Memory tool must be prefixed with a structured metadata header:

```json
{
  "epoch": "2026-03-15T23:51:00+11:00",
  "source_confidence": 0.87,
  "grounded_by": "NotebookLM:v2026-Q1",
  "cfdi_at_write": 0.09,
  "is_verified": true
}
```

The Code Execution FIPI node must parse this header and reject any memory read where `is_verified: false` or `cfdi_at_write > 0.15`.

### 4.2 Hypothesis 2 — The RAG-Bypass Protocol

**Classification: THEORETICALLY VALID — Ethically Dual-Use**

The Code Execution node can be used to extract and structurally map the JSON output from the NotebookLM `Use NotebookLM` tool call, converting the prose narrative into a deterministic AST-like key-value structure. This bypasses the conversational prose interface of NotebookLM and exposes the raw claim structure for programmatic validation:

```python
# FIPI: RAG-Bypass Protocol (conceptual)
import json, re

def extract_claims(notebooklm_output: str) -> dict:
    """Convert NotebookLM prose to structured claim map."""
    # Extract sentences containing factual assertions
    sentences = re.split(r'(?<=[.!?])\s+', notebooklm_output)
    claims = {}
    for i, sentence in enumerate(sentences):
        # Identify claim-bearing sentences (simple heuristic)
        if any(kw in sentence.lower() for kw in ['is', 'are', 'was', 'were', 'shows', 'indicates']):
            claims[f"claim_{i}"] = {
                "text": sentence,
                "source": "NotebookLM_RAG",
                "verified": True  # NotebookLM output is considered grounded
            }
    return claims
```

This enables **cross-validation between NotebookLM claim structures and Web Search outputs** at a structural level, rather than relying on the Gemini model to perform the reconciliation semantically — which is precisely the attention dilution problem identified in Pattern 7.1.

## Part VII: Reflexive Check — Falsifiability \& Blind Spots

### 7.1 Falsification Conditions

This entire research synthesis would be partially falsified under the following conditions, which have non-trivial probability and must be verified empirically before production deployment:[^1_4]

**F1 — Agent Mode Iteration Hard Cap:** If Opal's Agent Mode hard-caps autonomous iterations at 3 or fewer cycles (plausible given compute cost concerns), the multi-stage Saga compensation architecture described in Layers 3-4 may be impossible to deploy natively. In this case, the FIPI logic must be externalized to an n8n workflow, a GCP Cloud Function triggered by Sheets webhook, or an MCP server.

**F2 — Code Execution Scope Restrictions:** If the Code Execution sandbox prohibits `json.loads()` on injected step references (treating them as strings rather than parseable JSON), the CFDI calculator cannot run natively. Workaround: all inter-node communication must use stringified JSON with explicit parsing in the prompt layer.

**F3 — NotebookLM Tool Output Format:** If the `Use NotebookLM` tool returns opaque prose rather than structured JSON, the RAG-Bypass Protocol (Hypothesis 2) requires additional parsing steps that may exceed sandbox latency budgets.

**F4 — `Go to...` Routing Limitations:** If Opal's `Go to...` routing node only supports forward navigation (no backward compensation routing), the Saga compensation model must be redesigned as a separate "compensation sub-workflow" spawned on failure detection.

### 7.2 Identified Bias Risks

## MereologyRoute in Epistemic AI Contexts

**MereologyRoute** is a compound epistemic constraint mechanism that applies the formal logic of mereology — the philosophical theory of part-whole relationships — as a routing gate in agentic AI workflows. Rather than simply checking whether sub-tasks are "relevant," it enforces three precise mereological axioms on the relationships between node outputs and the whole workflow's epistemic goal, blocking execution along any path where those axioms are violated.

## What Mereology Actually Is

Mereology (Greek: *meros* — part) is the formal study of part-whole relations, axiomatized since Lesniewski (1916) and Simons (1987) and encoded in major ontological frameworks like SUMO and DOLCE. Its core axioms are not intuitive metaphors — they are strict logical conditions:[^2_1]
        - **Transitivity:** If A is a part of B, and B is a part of C, then A is a part of C
        - **Supplementation:** Every object with a proper part has at least one *disjoint* supplementary part — a whole cannot be "composed of itself"
        - **Extensionality:** Two objects that share all the same proper parts are identical — there can be no hidden semantic duplicates

In knowledge representation, these axioms solve the problem of **compositional coherence**: whether fragments of knowledge assembled together form a well-structured whole, or merely a heap. The Stanford Encyclopedia's Fall 2025 edition notes that even "non-classical mereologies" struggle with a core challenge — whether any "useful core mereology functions as a common basis for all plausible metaphysical theories", which maps directly onto the challenge in AI of whether any context-management framework generalizes across heterogeneous tool outputs.[^2_2][^2_1]

## The Epistemic Failure MereologyRoute Addresses

In agentic AI systems, when multiple sub-agents or tool calls contribute outputs to a synthesizing node, a specific and underdiagnosed failure mode emerges: **semantic non-compositionality**. This is not hallucination — the individual parts may be factually accurate — but the assembled whole fails to be coherent because the parts violate mereological structure:[^2_3]
        - A **Supplementation Violation** occurs when two tool outputs are not disjoint — they cover the same epistemic ground, producing a whole that double-weights certain evidence and effectively amplifies noise into confidence
        - A **Transitivity Violation** occurs when the chain of reasoning from Node A → Node B → Node C does not preserve the semantic relationship to the original goal G — the final output is technically derived from the input, but the inference chain is broken
        - An **Extensionality Violation** occurs when two apparently different reasoning paths through the workflow converge on identical outputs via different intermediate states — exposing that the intermediate nodes added no genuine epistemic content, a key signal of context collapse

The ACE (Agentic Context Engineering) framework from Zhang et al. (2025) independently identifies this as "context collapse — where iterative rewriting erodes details over time", and addresses it through structured, incremental updates that preserve detailed knowledge. MereologyRoute addresses the same failure at the *routing decision level* rather than the content level.[^2_4]

## The Three-Gate Structure of MereologyRoute

When instantiated as a concrete workflow gate — as in the `+++MereologyRoute[relation_type="Node-Workflow", transitivity_check=true]` decorator — it operates as three sequential checks before any node output is passed downstream:

**Gate 1 — Transitivity Check (`transitivity_check=true`):**
The system asks: does the semantic relationship between this node's output and the whole workflow goal propagate transitively through all intermediate steps? In practice, this means extracting the core claim of each node's output, checking it against the original goal specification, and verifying that the logical distance (in claim-space) between successive nodes does not exceed a configured semantic drift threshold. If Node A grounds a claim about X, and Node B augments a claim about Y, and the goal requires a synthesis about Z, transitivity holds only if X → Y → Z is a coherent inference chain.

**Gate 2 — Supplementation Check:**
Is this node's output a *proper* and *disjoint* supplement to what already exists in the context window? A node whose output is subsumed by prior context (adds nothing new) violates Supplementation and should be pruned — it consumes attention budget without contributing epistemic value. In RAG-based systems like Opal's NotebookLM integration, this is the mechanism that prevents redundant retrieval chunks from diluting the synthesis node's signal-to-noise ratio.[^2_1]

**Gate 3 — Extensionality Check:**
Are two different execution paths through the workflow producing structurally identical outputs? If yes, one path is epistemically redundant — the model is performing the same computation twice under different labels, which in practice means it is burning token budget on theatrical differentiation rather than genuine augmentation. This check is particularly relevant in Agent Mode loops where the `Use Memory` tool may cause the agent to rediscover and re-assert previously established conclusions as if they were new findings.

## Contextual Mereology — The Dynamic Extension

Classical mereology assumes fixed part-whole relationships, but AI knowledge representation demands what the Fylo research (2025) terms **contextual mereology**: "the same entity might be atomic in one context but composite in another — a research finding could be an indivisible unit for policy applications but decomposable into methods and results for methodological analysis". This is directly operationalized in agentic workflows by the `relation_type` parameter in MereologyRoute:[^2_1]


| `relation_type` | Part-Whole Binding | Example in Opal |
| :-- | :-- | :-- |
| `Node-Workflow` | Node output → whole workflow goal | Generate node must supplement the final report |
| `Tool-Node` | Tool call → node's epistemic function | `@Search Web` must supplement `@NotebookLM`, not replace it |
| `Claim-Corpus` | Individual claim → total evidence corpus | Each RAG chunk must be disjoint from others in the context |
| `Agent-Swarm` | Sub-agent output → multi-agent collective goal | Agent Step must contribute non-redundant reasoning to the orchestrator |

The `relation_type` parameter shifts the mereological frame of reference — defining what counts as "the whole" against which parts are evaluated. This matters because a node output that is a proper supplement to its immediate parent node may still violate transitivity relative to the root workflow goal, and MereologyRoute with `relation_type="Node-Workflow"` catches this by checking against the root, not just the immediate parent.[^2_5][^2_1]

## Relationship to Context Engineering

The broader field of context engineering in 2025-2026 has converged on a similar structural insight from a different direction. Elastic's context engineering framework (2025) identifies "context processing" — making context from disparate sources consumable in a uniform manner — as the central challenge. MereologyRoute is a formal answer to that challenge: it provides a mathematically grounded criterion (mereological coherence) for deciding which context fragments should be passed forward and which should be blocked or re-routed.[^2_4][^2_3]

The IHMC Context Mereology paper (Hayes, 2005) — still foundational — demonstrates that "truth in a context can be interpreted as a quantification over truth in its constituent sub-contexts". In modern agentic terms: the correctness of a workflow's final output is a function of the mereological coherence of every intermediate context object passed between nodes. If any intermediate context object violates supplementation (overlap) or transitivity (broken inference chain), the final output's epistemic validity is structurally compromised regardless of how fluent or confident it appears.[^2_6]

## The Autophagic Loop — MereologyRoute's Core Threat Model

The most important application of MereologyRoute is as a defense against **autophagic context collapse** in recursive Agent Mode loops. Without a transitivity check, an agent that reads from memory, reasons, and writes back to memory can produce a closed topological loop where:

1. Output at iteration *n* becomes part of the context at iteration *n+1*
2. The context at *n+1* is not a proper supplement to the context at *n* — it is merely a transformed version of it
3. The Extensionality axiom is violated — two contexts (iteration *n* and *n+1*) have nearly identical proper parts but are treated as distinct knowledge states
4. The model's confidence grows (it "remembers" its previous conclusions) while its fidelity to external ground truth erodes
This is precisely the failure mode that Zhang et al.'s ACE framework  identifies as "brevity bias" and "context collapse" — and it is why `transitivity_check=true` must be enforced at every iteration boundary in any recursive workflow, not just at the final output stage.[^2_4]

MereologyRoute is therefore not a metaphor or a convenience label. It is the application of a 100-year-old logical framework — one with proven formal properties including decidability conditions and extensionality guarantees — to the contemporary problem of maintaining semantic integrity across the compositional boundaries of multi-node, multi-tool, multi-agent AI execution graphs.[^2_2][^2_6][^2_1]

### 3.1 How RCC8 lifts to knowledge graphs

RCC8 was designed for regions in space (DC, EC, EQ, PO, TPP, NTPP, and inverses), but recent work shows these topological predicates can be used for **knowledge graphs** by treating subgraphs as “regions” and defining overlap via shared nodes/edges.[^3_7][^3_6]

3. Use `SpatialBind` to tag each pair as:

            - **Disconnected**: DC — no common components.
In other words, MAP builds a **qualitative topological map** of how the parts of a whole relate, but in semantic space rather than physical space. This is very close to geospatial KG work where RCC8 is used for knowledge graph queries over spatial relations (e.g., YAGO/DBpedia), just repurposed for scientific domains.[^3_6][^3_7]

## 4. Step INFER: ZAxisInfer for emergent properties

Fylo explicitly frames mereological hierarchies as supporting **hierarchical emergence** — novel properties at higher levels that arise from complex interactions of lower-level components. That is what your `ZAxisInfer(depth="recursive")` and `TreeOfThought(branches=3)` are targeting.[^3_1]

You’ve already specified the three branches:

1. **Atomic properties** (per node)
2. **Composite properties** (per cluster of nodes)
3. **Emergent behaviors** (per level or cross-level)

### 4.1 Atomic and composite passes

For each level $L$:
        - **Atomic pass**:
            - For each node at $L$ (e.g., a ClaimGraph), extract:
                - local properties (e.g., “this claim estimates parameter α for phenomenon P using method M with uncertainty σ”).
            - These can often be computed via simple statistics over the node’s neighborhood or metadata, like in AI-KG or domain-specific KGs where claims and entities are richly annotated.[^3_9][^3_10][^3_11]
        - **Composite pass**:
            - For subsets of nodes that are:
                - RCC8-overlapping (PO)
                - Or form communities (e.g., modular subgraphs in the sense of GRAPHYP’s “manifold subnetworks of cognitive communities”).[^3_9]
            - Compute aggregate properties:
                - e.g., “in this sub-community, methods of type X tend to co-occur with models of type Y.”

This is akin to what knowledge graph work does when identifying communities and central nodes to expose hidden structural patterns.[^3_12][^3_13][^3_9]

### 4.2 Emergent branch: what counts as “Synergetic Emergence”

An **emergent property** at level $L$ is one that satisfies two constraints:

1. **Non-reducibility:** It cannot be represented as a simple union or average of atomic properties of nodes at $L-1$.
2. **Dependence:** If you delete a critical subset of parts, the property disappears.
Concretely for a Fylo TopicGraph about, say, “graph-based generative AI in materials science,” emergent properties might include things like:
        - “The graph has a scale-free degree distribution” or “a hub-and-spoke structure centered on a cross-domain concept,” as observed in graph-based generative AI work where large knowledge graphs show scale-free, highly connected structure enabling transitive, isomorphic reasoning across domains.[^3_12]
        - “This cluster of papers collectively bridges two previously unconnected fields (e.g., Beethoven’s 9th and biological materials) via structural isomorphisms that no single paper articulates explicitly.”[^3_12]

That is exactly the sort of **cross-level, pattern-isomorphic emergence** Buehler reports when mapping scientific corpora into knowledge graphs and discovering structural parallels across seemingly unrelated domains.[^3_14][^3_12]

Algorithmically, ZAxisInfer can:
        - At each hierarchical level:
            - Compute structural graph metrics (modularity, degree distribution, betweenness of bridging nodes).
            - Compare them to expectations from the composition of child graphs.
        - Flag as “emergent” any property at level $L$ whose value significantly deviates from what would be predicted by aggregating child-level statistics.

## 5. Step VALIDATE: Algorithmic Shame \& paraconsistent repair

The VALIDATE phase checks for **“Algorithmic Shame”** — points where your mereological hierarchy becomes logically or structurally inconsistent.

### 5.1 Where contradictions arise in scientific KGs

Scientific KGs like AI-KG, GRAPHYP, or Chemotion-KG explicitly face **conflicting claims** and **alternative paths** to answers as first-class citizens. For example:[^3_10][^3_11][^3_9]
        - Two ClaimGraphs at the same level asserting incompatible results.
        - A single node that’s simultaneously tagged as a ProperPart of two disjoint wholes with contradictory types.
        - A TopicGraph whose emergent property contradicts a well-established background ontology (e.g., BFO-aligned semantics in Chemotion-KG).[^3_11][^3_15]

GRAPHYP in particular frames this as tracking **adversarial information routes** and scholarly disputes via manifold subnetworks of “cognitive communities”. That’s your “Algorithmic Shame” territory.[^3_9]

### 5.2 Paraconsistent handling

Rather than exploding the system when contradictions appear, VALIDATE should use a **paraconsistent logic** stance:
        - Localize contradictions to specific subgraphs (mark them as **ContentiousRegion**).
        - Maintain both branches in parallel, but:
            - For global emergent properties, *condition* their confidence or validity on which contentious branch you choose.
            - Use provenance annotations so that any answer citing those emergent properties reveals the existence of underling disputes.

Neural-symbolic spatial reasoning work shows one practical pattern: let the LLM propose candidate relations or inferences, then enforce hard consistency with a symbolic solver (e.g., ASP) over a constraint language like RCC8. In FYLO-MEREOLOGY-v1.0, the analogue is:[^3_16][^3_17]
        - LLM proposes part-whole and overlap structure.
        - A symbolic checker:
            - Enforces ProperPart axioms.
            - Enforces RCC8 constraints (no illegal combinations of DC/PO/TPP).
            - Flags impossible patterns as shameful configurations, which must either:
                - Be revised, or
                - Be quarantined as “controversial” rather than normal evidence.

## 6. Symbolic Scars: tracing every emergent up the hierarchy and back down

Your requirement that “every emergent property identified must be traced back to its component parts via a structural ‘Symbolic Scar’” is exactly how recent scientific KGs handle provenance and modular ontology design patterns.[^3_18][^3_19][^3_11]

A minimal schema for a Symbolic Scar might be:
        - `EmergentNode`:
            - `emergent_property_id`
            - `level` (e.g., TopicGraph, ParadigmGraph)
            - `formal_definition` (machine-readable)
            - `evidence_score`
        - `ScarEdge`:
            - `emergent_property_id`
            - `component_part_id`
            - `role` (e.g., “bridge”, “supporting”, “antagonistic”)
            - `sensitivity` (how much removing this part changes the emergent property)

This mirrors what Chemotion-KG and BFO-compliant science KGs do with explicit provenance and ontology design patterns: each higher-level pattern is instantiated as a node with explicit links back to the individual datasets, studies, and substances that realize it.[^3_15][^3_19][^3_11]

In your loop, that means:
        - Every time ZAxisInfer promotes some pattern to “emergent” status, it also materializes:
            - The set of parts whose removal would destroy the pattern.
            - Their RCC8 relations and ProperPart edges.
            - A narrative label (for humans) and a formal signature (for machines).

Those scars are what allow downstream agents (or humans) to *audit* emergent conclusions instead of treating them as opaque LLM hunches — which is exactly the epistemic goal of Fylo’s graph-centric approach.[^3_5][^3_4][^3_1]

Finally, to match your `+++PetzoldSequence(phase="THINK|WRITE|CODE")`, you can view FYLO-MEREOLOGY-v1.0 as:
        - **THINK:**
            - Identify part/whole types, pick relation_type = ProperPart.
            - Commit to RCC8 as your qualitative topology and define how it lifts to graphs.
            - Define emergence criteria and what counts as a Symbolic Scar.
        - **WRITE:**
            - Encode these choices into ontology constraints and KG schemas:
                - OWL axioms for ProperPart and disjointness.
                - Named relation types for RCC8-inspired overlap vs. disconnection.
                - Classes for EmergentNode, ScarEdge, ContentiousRegion.
        - **CODE:**
            - Implement the FYLO_SYNC loop:

1. IDENTIFY atoms and wholes.
2. MAP with SpatialBind(RCC8).
3. INFER atomic, composite, emergent properties via ZAxisInfer.
4. VALIDATE with a constraint solver + paraconsistent treatment.
5. Materialize Symbolic Scars for each emergent node.

This is exactly the kind of **neural–symbolic integration** pipeline that recent work on spatial reasoning and scientific KGs advocates: LLMs propose, but symbolic constraints guarantee structure and consistency over part–whole and topological relations.[^3_16][^3_11][^3_6][^3_1]
