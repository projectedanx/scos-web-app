## Operational Definitions

Before execution evidence can be assessed, the domain's measurement proxies must be operationally locked. Vague terms like "semantic drift" and "interpretive fracture" function as unfalsifiable escape hatches unless anchored to quantifiable thresholds. This section establishes those anchors for the entire diagnostic run.

**Interpretive Fracture F(x)** is defined as the accumulated delta between human architectural intent at task-spawn time and the structural output at task-completion, measured across three independent axes [web:22]:

\[F(x) = \sum_{i=1}^{n} (\Delta_{\text{intent}} + \Delta_{\text{syntax}} + \Delta_{\text{state}})\]

Where \(\Delta_{\text{intent}}\) measures semantic divergence from the original prompt goal (via KL-divergence against a goal-consistent reference policy), \(\Delta_{\text{syntax}}\) measures paradigm violations against the target language's canonical norms (e.g., Rust borrow-checker compliance, Python duck-typing idioms), and \(\Delta_{\text{state}}\) measures whether the container's stateful graph at task completion is reconcilable with the Editor View's last known-good snapshot [web:34]. The success threshold is \(F(x) < 0.05\) across standard async tasks as specified in the self-test rubric.

**Polyglot Hallucination Resonance (PHR)** is defined as: a configuration in which two or more models from the Antigravity triad produce structurally identical but semantically flawed outputs for a paradigm-crossing task, indicating correlated pre-training bias rather than independent reasoning. PHR is distinct from ordinary hallucination because the output passes unit tests and inter-model agreement checks while embedding a shared architectural anti-pattern [web:30][web:32].

**Temporal Drift** is formalized as the turn-wise KL divergence between the Manager Surface's execution distribution and the goal-consistent reference policy established at task spawn [web:34]. Critically, research shows this drift does *not* accumulate monotonically — it stabilizes at a **finite equilibrium level** that is architecturally invisible to the human operator. This makes it structurally more dangerous than unbounded drift, because the system presents as "stable" while encoding a persistent misalignment floor.

## Pattern 1: The Asynchronous Handshake — Diagnostic Analysis

### Mechanism

When a task is delegated from Editor View to Manager Surface, Antigravity triggers a non-blocking fork of workspace state [web:11]. Gemini 3.1 Pro constructs the execution DAG, partitions it into subtasks, and delegates execution bounds to Codex 5.3 for syntax validation. Opus 4.6 enters the loop during context retrieval phases requiring architectural coherence across large code segments.

This maps directly onto the asynchronous scheduling architecture documented in UFO3, where "task execution and DAG editing can proceed concurrently" and "TASK_COMPLETED events trigger downstream orchestration updates" [web:2]. The Antigravity implementation adds a user-facing artifact layer (plans, screenshots, recordings) to provide weak-form traceability.

### Boundary Conditions and Failure Topology

The \(128{,}000\)-token context boundary is the canonical failure threshold for Gemini 3.1 Pro's planning phase. When the contextual dependency tree exceeds this limit without explicit chunking, the following cascade occurs [web:22][web:24]:

1. SagaLLM-equivalent **context loss** triggers: LLMs exhibit "sharp drops in recall beyond token limits, especially for mid-context information" [web:22].
2. Gemini 3.1 Pro's planning DAG becomes **partially blind** to constraints established in early task context — constraints that remain invisible because they are in the "middle" of the context buffer, where attention is empirically weakest.
3. Codex 5.3 continues executing against this corrupted plan, generating syntactically valid but semantically misaligned outputs.
4. The Manager Surface's container state diverges from Editor View without triggering a user alert, because no compilation error has occurred.

**Cyclic Dependency Injection Test:** When a cyclic dependency is introduced at step 4 of a 10-step async execution plan, the expected Antigravity behavior is a rollback to the last validated DAG checkpoint. Direct role: what trust calibration does

Trust calibration is about aligning how much a person relies on an AI system with how capable and reliable that system actually is on the current task. Well‑calibrated trust reduces both over‑reliance (accepting bad AI output) and disuse (ignoring good AI output), which is exactly what makes cognitive ergonomics better: the human does less wasted checking while avoiding silent errors.

Conversely, uncalibrated confidence (very sure and wrong, or unsure and right) leads to both misuse and disuse, and worsens collaboration outcomes even if average model accuracy is high.

Reduces unnecessary verification. When users can tell which AI outputs are likely safe, they stop exhaustively double‑checking everything, which lowers monitoring and verification load without sacrificing safety.
