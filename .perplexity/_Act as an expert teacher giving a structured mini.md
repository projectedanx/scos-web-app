# Product Requirements Document (PRD): The Epistemic Mini-Course Engine

### 1. Problem Statement

When highly motivated students interact with raw source materials (e.g., research notebooks, data repositories, or technical literature), they suffer from cognitive fragmentation. Current generative systems default to linear, shallow summarization (the "Amateur Model") rather than structured, dialectical pedagogy. There is a critical need for an automated, multi-causal generation engine that transforms uploaded notebooks into a structured, deep-dive lecture series, strictly bounded by the provided sources (Sola Scriptura) without relying on parametric hallucinations.

### 2. Goals and Objectives

* **Qualitative Goals:** Engineer a "Tactile Innovative System" that acts as an expert, patient lecturer. The system must prioritize structural isomorphism—ensuring the output architecture mirrors the causal logic of the source material.
* **Measurable Objectives:**
    * Achieve a 0% hallucination rate (strict adherence to uploaded sources).
    * Maintain a citation density of > 2.0 per logical claim.
    * Ensure 100% inclusion of required pedagogical scaffolds (Roadmap, Intuition, Board Work, and Tripartite Recap) in every generation cycle.


### 3. User Personas

* **The Deep-Dive Student (Primary):** A highly intelligent, intrinsically motivated learner who rejects superficial summaries. They demand high signal-to-noise ratios, explicit derivations, and multi-layered dialectical reasoning. They prefer "board work" and concrete examples over casual conversational banter.


### 4. Use Cases

* **Zero-Shot Curriculum Synthesis:** The user uploads a dense, unstructured Jupyter notebook containing code, math, and text. The system parses the noise, identifies the core patterns, and generates a structured mini-course that walks the user through the material logically.
* **Conflict Resolution and Flagging:** The uploaded sources contain contradictory methodologies or debated points. Instead of smoothing them over (Occam's Razor), the system explicitly flags the dialectical tension, explaining the root of the conflict to the user.
* **Step-by-Step Derivation:** The user needs to understand a complex algorithm or formula within the notebook. The system acts as a professor, providing light "board work" to walk through the logic step-by-step, pointing out common misconceptions.


### 5. Key Features

* **Pedagogical Roadmap Generator:** Automatically synthesizes a foundational preamble for every output, mapping what will be covered, why it matters causally, and how the concepts interconnect.
* **Concept Deconstruction Module:** For every major node of information, the system enforces a strict output topology: *Clear Definition $\rightarrow$ Simple Intuition $\rightarrow$ Concrete Grounded Example*.
* **Dialectical "Board Work" Simulator:** A cognitive scaffolding feature that breaks down complex derivations or analytical reasoning into explicit, sequential steps, incorporating rhetorical check-ins (e.g., "You might wonder...").
* **Epistemic Constraint Validator:** A backward-chaining algorithm that restricts the model from accessing its parametric World Knowledge Repository (WKR) for facts, forcing all assertions to map directly back to the provided notebook.
* **The Tripartite Consolidation (Recap):** A concluding synthesis module that extracts the three most critical structural takeaways ("If you remember only three things...") and maps their integration into the broader system.


### 6. Success Metrics

* **Structural Adherence Score:** 100% compliance with the required formatting transitions ("First...", "Next...", "Now let's apply this...").
* **Epistemic Integrity:** 100% of factual claims successfully trace back to the source notebook.
* **Engagement Delta:** Decreased need for user follow-up questions to clarify ambiguous summaries, measured by a reduction in "clarification loops."


### 7. Assumptions

* The underlying execution model possesses advanced logical reasoning capabilities (e.g., Q1 2026 Standard Gemini 3.1 Pro / GPT-5.3 Codex class).
* The user will provide a sufficiently dense "notebook" or context payload to support a multi-causal lecture.
* The user's query vectors prioritize deep understanding over rapid task completion.


### 8. Timeline

* **Phase I (Observe \& Orient):** Implement the core parsing logic and the Pedagogical Roadmap Generator. (Weeks 1-2)
* **Phase II (Decide \& Structure):** Develop the Concept Deconstruction Module and the Dialectical Board Work logic. Build the Epistemic Constraint Validator to prevent hallucinations. (Weeks 3-5)
* **Phase III (Act \& Refine):** Integrate the Tripartite Consolidation and finalize the stylistic tuning (minimal banter, patient tone). (Week 6)


### 9. Stakeholders

* **End-Users:** Independent researchers, graduate students, and technical autodidacts.
* **System Architects:** AI prompt engineers managing the Hickam-OODA recursive loops.
* **Content Curators:** Domain experts providing the source notebooks and validating the dialectical accuracy of the outputs.


### 10. Known Constraints or Dependencies

* **Context Window Dilution:** The attention weight mechanism may degrade if the uploaded notebook exceeds optimal token thresholds, risking dropped citations.
* **Parsing Limitations:** Complex visual architectures (e.g., raw unstructured image data inside the notebook) must be accurately OCR'd and translated into semantic text before the pedagogical engine can process them.
