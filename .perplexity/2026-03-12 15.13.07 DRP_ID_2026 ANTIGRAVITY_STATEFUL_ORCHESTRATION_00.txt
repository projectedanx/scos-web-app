The Unit Test Trap

The most dangerous manifestation of Temporal Drift is that unit tests will typically pass for a drifted output. This is because Codex 5.3, as the syntax grounding engine, generates both the implementation and the tests within the same execution context — tests are written against what was built, not against what was intended. This creates a structural falsification problem: the standard CI/CD validation pipeline cannot detect Temporal Drift because it operates at the syntactic/functional layer, not the architectural-intent layer.

The boundary condition for safe async delegation is therefore not token count but architectural stability depth — the number of cross-module dependency changes required to implement the delegated task. Tasks with high architectural stability depth (refactoring a stateful service to a functional-reactive paradigm; migrating from synchronous I/O to async across 40 files) have high Temporal Drift risk. Tasks with low architectural stability depth (adding a utility function, writing documentation, generating tests for existing stable interfaces) have low Temporal Drift risk and are the genuine target use case for Antigravity's async delegation.
